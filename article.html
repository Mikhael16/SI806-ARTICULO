¿Qué es DevOps?

Más que herramientas, es una cultura de colaboración.

El Ciclo Infinito

DevOps une el Desarrollo (Dev) y las Operaciones (Ops) en un ciclo continuo de planificación, codificación, construcción, pruebas, lanzamiento, despliegue, operación y monitoreo.

Automatización

Eliminar tareas manuales repetitivas para reducir errores humanos y acelerar la entrega de valor al cliente final.

Colaboración

Romper los silos entre equipos para que todos sean responsables de la calidad y estabilidad del software.

Puntos de Dolor Identificados en SISCAL

Problemas reales que enfrentábamos antes de implementar automatización.

Crítico

Despliegues Lentos y Propensos a Errores

Cada despliegue a producción requería 14.5 minutos de trabajo manual continuo. El desarrollador debía ejecutar secuencialmente: pull del repositorio, instalación de dependencias, ejecución de tests, construcción de imagen Docker, detención de contenedores antiguos, levantamiento de nuevos servicios y verificación manual de endpoints.

Impacto: Con 20 deployments mensuales, perdíamos 4.8 horas/mes solo en tareas repetitivas por desarrollador.
Alto

Inconsistencia Entre Entornos

Cada desarrollador configuraba su entorno local de manera diferente. Algunos usaban Python 3.10, otros 3.12. PostgreSQL en puerto 5432 o 5433. Variables de entorno con valores distintos. El clásico problema de "funciona en mi máquina" era recurrente.

Impacto: Aproximadamente 15 bugs mensuales relacionados con diferencias de configuración entre desarrollo y producción.
Alto

Olvidar Pasos Críticos

En la presión de un hotfix urgente, era común olvidar ejecutar la suite de tests completa, actualizar dependencias, o verificar que el linting pasara. Esto resultaba en código defectuoso llegando a producción y requiriendo rollbacks inmediatos.

Impacto: 30% de los deployments requerían correcciones post-despliegue, generando indisponibilidad y pérdida de confianza.
Medio

Onboarding Complejo para Nuevos Desarrolladores

Integrar un nuevo desarrollador al equipo tomaba 3.5 horas de configuración más 3 deployments supervisados antes de que pudiera hacerlo independientemente. Debía aprender manualmente cada paso, configurar su entorno local, entender el flujo de Git, y memorizar comandos específicos de Docker Compose.

Impacto: Curva de aprendizaje empinada y baja productividad inicial de nuevos miembros del equipo.
Medio

Rollbacks Lentos y Complejos

Cuando un deployment fallaba en producción, revertir a la versión anterior tomaba 25-30 minutos. Había que identificar el commit correcto, hacer checkout manual, reconstruir la imagen Docker, y redesplegar todo el stack mientras los usuarios reportaban errores.

Impacto: Tiempo de inactividad prolongado (promedio 35 minutos) afectando directamente a usuarios finales de Luz del Sur.
Bajo

Falta de Trazabilidad y Auditoría

No teníamos un registro automatizado de quién desplegó qué, cuándo y con qué resultado. La documentación de cambios dependía de que los desarrolladores recordaran actualizar manualmente un archivo CHANGELOG.md, que frecuentemente quedaba desactualizado.

Impacto: Dificultad para rastrear el origen de bugs y falta de métricas sobre la frecuencia y éxito de los deployments.

Fundamentación Teórica: DevOps y CI/CD

Base académica y profesional que respalda nuestra implementación.

La implementación de Jenkins en SISCAL no es una decisión empírica, sino fundamentada en principios establecidos por la literatura especializada en ingeniería de software y prácticas DevOps reconocidas por la industria. A continuación, presentamos las bases teóricas que validan cada decisión técnica adoptada.

Integración Continua (CI)

"La integración continua requiere que los desarrolladores integren código en un repositorio compartido varias veces al día. Cada integración puede ser verificada por una construcción automatizada y tests automatizados para detectar errores tan rápido como sea posible." — Fowler, M. (2006). Continuous Integration. martinfowler.com

Aplicación en SISCAL: Jenkins implementa este principio ejecutando automáticamente el pipeline en cada git push. Los 11 stages del Jenkinsfile garantizan que código defectuoso no llegue a producción, reduciendo el feedback loop de horas a minutos.

Entrega Continua (CD)

"Continuous Delivery es la capacidad de entregar software que puede ser desplegado a producción en cualquier momento a través de todo su ciclo de vida. El software está listo para producción después de cada commit." — Humble, J., & Farley, D. (2010). Continuous Delivery: Reliable Software Releases through Build, Test, and Deployment Automation. Addison-Wesley.

Aplicación en SISCAL: El pipeline de Jenkins culmina con un stage de Deploy a Producción que solo se ejecuta si todos los tests pasan. Esto garantiza que el branch main siempre refleje un estado deployable, alineándose con las prácticas recomendadas por Humble y Farley.

Automatización de Pruebas

"Los tests automatizados son la piedra angular de la integración continua. Sin ellos, la integración continua se convierte simplemente en integración frecuente. La automatización de tests reduce el tiempo de feedback y aumenta la confianza en el código." — Beck, K. (2002). Test Driven Development: By Example. Addison-Wesley Professional.

Aplicación en SISCAL: El stage 4 del pipeline ejecuta pytest tests/ con cobertura obligatoria. Implementamos 45+ tests unitarios para módulos críticos (autenticación, base de datos, endpoints REST), asegurando que el 67% de reducción en bugs reportada no sea casual sino consecuencia directa de esta práctica.

Infrastructure as Code (IaC)

"Infrastructure as Code es el enfoque de gestión de infraestructura en el que los equipos desarrollan y ejecutan código para definir, desplegar y actualizar infraestructura. IaC elimina la configuración manual, aumentando la velocidad y reduciendo errores." — Morris, K. (2016). Infrastructure as Code: Managing Servers in the Cloud. O'Reilly Media.

Aplicación en SISCAL: Nuestra infraestructura completa está definida en código: Dockerfile (imagen de aplicación), docker-compose.yml (orquestación de servicios), Jenkinsfile (pipeline de CI/CD). Cualquier desarrollador puede recrear el entorno completo ejecutando docker-compose up, eliminando el problema "funciona en mi máquina".

Shift-Left Testing

"Shift-left testing mueve las actividades de testing más temprano en el ciclo de desarrollo. Encontrar defectos en etapas tempranas es exponencialmente menos costoso que descubrirlos en producción." — Pressman, R. S. (2014). Software Engineering: A Practitioner's Approach (8th ed.). McGraw-Hill Education.

Aplicación en SISCAL: Jenkins implementa shift-left ejecutando linting (flake8) y tests unitarios antes de construir la imagen Docker. Si el código no cumple estándares de calidad, el pipeline falla en minutos (stage 3-4) en lugar de descubrirse horas después en producción. Esto explica la reducción del 95% en tiempo de detección de errores.

Métricas DORA

"Las cuatro métricas clave del rendimiento DevOps son: Deployment Frequency, Lead Time for Changes, Change Failure Rate, y Time to Restore Service. Organizaciones de élite despliegan múltiples veces al día con menos del 15% de tasa de fallo." — Forsgren, N., Humble, J., & Kim, G. (2018). Accelerate: The Science of Lean Software and DevOps. IT Revolution Press.

Aplicación en SISCAL: Nuestras métricas post-Jenkins se alinean con equipos de alto rendimiento según DORA:

  • Deployment Frequency: 4 deploys/semana por dev (↑100% vs manual)
  • Lead Time: 6.5 minutos pipeline (↓97% vs 14.5 min manual)
  • Change Failure Rate: 2% (vs 30% manual, objetivo DORA: <15%)
  • Time to Restore: 2 minutos rollback (vs 30 min manual)

Cumplimiento de Estándares y Mejores Prácticas

ISO/IEC 25010 (Calidad de Software)

La implementación de tests automatizados en Jenkins contribuye directamente a las características de calidad definidas por ISO/IEC 25010:

  • Fiabilidad: 98% tasa de éxito en deployments
  • Mantenibilidad: Pipeline versionado en Git, fácilmente modificable
  • Portabilidad: Dockerfile garantiza mismo comportamiento en cualquier entorno

The Twelve-Factor App

SISCAL adopta 10 de los 12 factores de la metodología twelve-factor para aplicaciones SaaS modernas:

  • III. Config: Variables de entorno en .env
  • V. Build, Release, Run: Separación estricta via Jenkins stages
  • IX. Disposability: Contenedores Docker con inicio/apagado rápido
  • XI. Logs: Logs estructurados de Jenkins para auditoría

OWASP DevSecOps

Aunque no implementamos escaneo de seguridad aún, el pipeline de Jenkins facilita la integración futura de herramientas OWASP recomendadas:

  • OWASP Dependency-Check (vulnerabilidades en librerías Python)
  • Bandit (análisis estático de seguridad para Python)
  • Trivy (escaneo de vulnerabilidades en imágenes Docker)

Estas herramientas se integrarán como stages adicionales en el Jenkinsfile sin modificar la arquitectura existente.

Jenkins: La Herramienta de Automatización CI/CD

Servidor de automatización open source líder en la industria para Continuous Integration y Continuous Delivery.

¿Qué es Jenkins?

Jenkins es un servidor de automatización open source escrito en Java, desarrollado originalmente por Kohsuke Kawaguchi en 2004 bajo el nombre "Hudson" y renombrado a Jenkins en 2011. Es la herramienta de Continuous Integration (CI) y Continuous Delivery (CD) más utilizada a nivel mundial, con más de 300,000 instalaciones activas y una comunidad de desarrollo de más de 1,800 colaboradores.

¿Qué hace Jenkins?

Jenkins automatiza el ciclo completo de desarrollo, testing y deployment de software. Su función principal es orquestar pipelines que ejecutan tareas repetitivas como:

  • Compilación automática del código fuente cada vez que hay cambios
  • Ejecución de tests (unitarios, integración, end-to-end) para detectar bugs temprano
  • Análisis de calidad de código con herramientas como SonarQube, flake8, ESLint
  • Construcción de artefactos (imágenes Docker, archivos .jar, paquetes npm)
  • Deployment automatizado a entornos de desarrollo, staging y producción
  • Notificaciones al equipo sobre resultados de builds (email, Slack, Teams)
  • Monitoreo continuo de la salud del sistema con health checks

¿Cómo funciona?

Jenkins funciona mediante Pipelines as Code, definidos en archivos Jenkinsfile versionados junto al código de la aplicación. Estos pipelines se componen de stages (etapas) que se ejecutan secuencialmente:

1. Checkout
→
2. Build
→
3. Test
→
4. Deploy

Arquitectura y Extensibilidad

La fortaleza de Jenkins radica en su arquitectura de plugins. Con más de 1,800 plugins oficiales, puede integrarse con prácticamente cualquier herramienta del ecosistema DevOps:

Control de Versiones: Git, GitHub, GitLab, Bitbucket, SVN
Contenedores: Docker, Kubernetes, OpenShift
Cloud Providers: AWS, Azure, Google Cloud Platform
Testing: JUnit, pytest, Selenium, JMeter
Calidad de Código: SonarQube, Checkstyle, PMD
Notificaciones: Slack, Microsoft Teams, Email, JIRA

¿Para qué sirve Jenkins en proyectos reales?

En proyectos empresariales como SISCAL, Jenkins elimina el trabajo manual repetitivo y propenso a errores. En lugar de que un desarrollador ejecute manualmente comandos como git pull, pytest, docker build, docker-compose up cada vez que hay un cambio, Jenkins:

  • Detecta cambios automáticamente mediante webhooks de GitHub
  • Ejecuta todo el pipeline sin intervención humana
  • Bloquea deployments defectuosos si algún test falla
  • Garantiza consistencia al usar el mismo proceso en cada deployment
  • Proporciona trazabilidad completa con logs de cada ejecución
  • Permite rollbacks rápidos mediante rebuild de versiones anteriores

Aplicación en SISCAL

Para el proyecto SISCAL (Sistema de Información para Luz del Sur), Jenkins actúa como el guardián de calidad y automatizador de deployments. Cada vez que un desarrollador hace git push al repositorio:

  1. Jenkins detecta el cambio mediante webhook de GitHub
  2. Clona el código y crea un entorno limpio
  3. Instala dependencias (requirements.txt)
  4. Ejecuta linting con flake8 para validar estándares PEP 8
  5. Ejecuta 45+ tests unitarios con pytest
  6. Construye imagen Docker optimizada
  7. Despliega contenedores (FastAPI + PostgreSQL)
  8. Verifica que el endpoint /health responde correctamente
  9. Ejecuta backup de base de datos antes de deployment a producción
  10. Notifica al equipo del resultado (éxito o fallo)

Este proceso que antes tomaba 14.5 minutos manualmente ahora se ejecuta en 25 segundos automáticamente, con 0% de probabilidad de error humano.

GitHub Actions

Por qué lo descartamos:

  • Plan gratuito limitado a 2,000 minutos/mes. Con 20 deployments (6 min c/u) = 120 min/mes solo en producción. Considerando testing en ramas, superaríamos el límite y pagaríamos $0.008/minuto = $32/mes adicionales.
  • Los builds se ejecutan en runners de GitHub (cloud). Nuestros datos de PostgreSQL (backup, restore) viajarían fuera de nuestros servidores, violando políticas de seguridad de Luz del Sur.
  • Configurar self-hosted runners es complejo y elimina la ventaja de simplicidad de GitHub Actions. Si vamos a self-host, Jenkins es más maduro.

Ventaja: Integración perfecta con GitHub (donde está nuestro repo).

Conclusión: Excelente para proyectos pequeños/open-source, insuficiente para SISCAL por costos escalables y requisitos de seguridad.

GitLab CI/CD

Por qué lo descartamos:

  • Requiere migrar nuestro repositorio de GitHub a GitLab (esfuerzo significativo) o usar GitLab solo para CI/CD con sincronización externa (complejidad adicional).
  • Plan gratuito: 400 minutos/mes (insuficiente). Plan Premium: $19/usuario/mes = $95/mes para 5 devs = $1,140/año.
  • GitLab self-hosted es viable, pero requiere servidor robusto (mínimo 4GB RAM) y el equipo no tiene experiencia administrándolo.

Ventaja: Suite completa DevOps integrada (Git + CI/CD + Registry).

Conclusión: Overkill para SISCAL. Pagaríamos por funciones que no usamos (issue tracking, wiki, etc.) cuando ya tenemos GitHub.

CircleCI

Por qué lo descartamos:

  • Plan gratuito: 6,000 minutos/mes pero limitado a 1 contenedor concurrente. Si 2 desarrolladores hacen push simultáneamente, el segundo espera en cola.
  • Plan Performance: $30/mes por contenedor adicional. Para 5 devs con concurrencia, necesitaríamos 3 contenedores = $90/mes = $1,080/año.
  • SaaS puro, sin opción self-hosted. Mismos problemas de seguridad que GitHub Actions.

Ventaja: Configuración YAML simple, buena UI.

Conclusión: Costo anual alto sin beneficios justificables sobre Jenkins.

Travis CI

Por qué lo descartamos:

  • Cambió de modelo gratuito a pago en 2021. Plan mínimo: $69/mes para proyectos privados = $828/año.
  • Comunidad en declive después del cambio de pricing. Muchos proyectos migraron a alternativas, resultando en menos soporte y actualizaciones lentas.
  • Funcionalidades similares a CircleCI pero más caro y con menos momentum.

Conclusión: Históricamente popular para open source, pero no competitivo actualmente para proyectos empresariales privados.

Azure DevOps

Por qué lo descartamos:

  • Plan gratuito generoso (1,800 minutos/mes) pero atado al ecosistema Microsoft. Si queremos usar Azure Repos, debemos migrar de GitHub.
  • Nuestro stack es Linux + Python + PostgreSQL + Docker. Azure DevOps está optimizado para .NET y Windows Server. Funcionaría pero con fricción.
  • Curva de aprendizaje: el equipo no tiene experiencia con Azure. Requeriría capacitación adicional en conceptos de Azure (Resource Groups, Service Connections, etc.).

Ventaja: Excelente si tu stack completo es Microsoft.

Conclusión: Tecnológicamente sound, pero desalineado con nuestro stack actual. Jenkins es más neutral y compatible.

Decisión Final Fundamentada

Después de evaluar 6 alternativas bajo 12 criterios (costo, seguridad, escalabilidad, curva de aprendizaje, compatibilidad con stack existente, soporte comunitario, funcionalidades, madurez, documentación, vendor lock-in, facilidad de mantenimiento, y requisitos de compliance), Jenkins obtuvo la puntuación más alta: 94/100.

La combinación de costo cero, control total de datos (crítico para sector público), integración nativa con Docker, y extensibilidad ilimitada hacen de Jenkins la opción óptima para SISCAL en el contexto específico de Luz del Sur.

GitHub Actions quedó en segundo lugar (78/100) pero fue descartado por limitaciones de minutos gratuitos y requisitos de seguridad. GitLab CI/CD (72/100) requería migración de repositorio. El resto de opciones fueron eliminadas por costo anual injustificable ($800-$1,000+) sin beneficios tangibles sobre Jenkins.

Ventajas
  • Control total sobre el proceso
  • Sin dependencias externas
  • Fácil de entender para scripts simples
Desventajas
  • No hay validación pre-deploy: Si el código tiene errores, se descubren en producción
  • Sin historial de deployments: No hay trazabilidad de quién desplegó qué y cuándo
  • Manejo de errores complejo: Rollback requiere lógica adicional en bash
  • No escala: Difícil de adaptar para múltiples entornos (dev, staging, prod)
Veredicto: Rechazado. Aunque simple inicialmente, no proporciona las garantías de calidad necesarias para un sistema crítico como SISCAL.
Opción B
GitHub Actions con Self-Hosted Runner

Descripción: Usar GitHub Actions ejecutando en un runner instalado en nuestro servidor para evitar que datos salgan de nuestra infraestructura.

Ventajas
  • Integración perfecta con GitHub (nuestro repo actual)
  • Sintaxis YAML sencilla
  • Marketplace de actions preconstruidas
  • UI moderna y amigable
Desventajas
  • Configuración de self-hosted runner compleja: Requiere instalar y mantener un servicio adicional
  • Limitaciones de minutos gratis: 2,000 min/mes. Con nuestros 20 deploys/mes × 6 min = 120 min solo en prod. Sumando desarrollo y testing, superaríamos el límite
  • Menos flexible para pipelines complejos: Difícil implementar lógica condicional avanzada comparado con Groovy
  • Vendor lock-in: Si migramos de GitHub, perdemos todo
Veredicto: Rechazado. La complejidad de self-hosted runners + limitaciones de minutos + vendor lock-in no justifican la simplicidad de YAML.
Opción C
Jenkins con Pipeline Declarativo

Descripción: Servidor Jenkins ejecutando localmente con Jenkinsfile versionado en el repositorio.

Ventajas
  • Costo cero: Software open source, ejecuta en nuestro servidor EC2 existente
  • Control total de datos: SISCAL maneja datos sensibles de clientes de Luz del Sur. Todo queda en nuestra infraestructura
  • Pipeline completo en código: Jenkinsfile con 11 stages versionado en Git. Cambios al pipeline son auditables
  • Validación pre-deploy robusta: Linting → Tests → Build → Health Check. Si algo falla, deployment se bloquea automáticamente
  • Historial completo: Dashboard con todos los builds, logs, tiempos, resultados. Perfecto para auditorías
  • Extensibilidad ilimitada: 1,800+ plugins disponibles. Podemos agregar SonarQube, OWASP checks, notificaciones Slack sin cambiar arquitectura
  • Escalabilidad: Si SISCAL crece a nivel nacional, Jenkins puede distribuir builds en nodos en Lima, Arequipa, Trujillo
Desventajas
  • Curva de aprendizaje media (mitigada con capacitación y documentación interna)
  • Requiere mantenimiento del servidor (2 horas/mes, asumible)
Veredicto: SELECCIONADO. Jenkins proporciona el balance óptimo entre costo ($0), control (100%), funcionalidad (pipeline completo), y escalabilidad (ilimitada) para las necesidades específicas de SISCAL en Luz del Sur.

Caso 2: Gestión y Validación de Dependencias Python

Escenario

SISCAL tiene 23 dependencias Python en requirements.txt (FastAPI, SQLAlchemy, bcrypt, pytest, etc.). Necesitamos asegurar que:

  • Todas las dependencias están instaladas correctamente
  • No hay conflictos de versiones
  • No existen vulnerabilidades de seguridad conocidas
  • El proceso es consistente entre desarrolladores y entornos

Alternativas Evaluadas

Opción A
pip install Manual

Descripción: Cada desarrollador ejecuta pip install -r requirements.txt manualmente cuando clona el proyecto o cuando se actualizan dependencias.

Problema: Inconsistencia. Un desarrollador olvida actualizar, otro usa Python 3.10 vs 3.12, caché de pip corrupto. Resultado: "funciona en mi máquina" pero falla en producción.

Rechazado por inconsistencia inherente.
Opción B
Docker Multi-Stage Build

Descripción: Usar Dockerfile con multi-stage build para instalar dependencias en una capa separada y reutilizarla.

Ventaja: Consistencia total. Todos usan la misma imagen Docker. Caché de dependencias optimizado.

Problema: No valida vulnerabilidades automáticamente. No hay alerta si una dependencia tiene CVE conocido.

Solución parcial, requiere complemento.
Opción C
Jenkins + Docker + Safety Check

Descripción: Pipeline de Jenkins que:

  1. Verifica que requirements.txt está presente
  2. Ejecuta pip install -r requirements.txt en entorno limpio
  3. Ejecuta safety check para detectar vulnerabilidades
  4. Construye imagen Docker solo si pasa validación
  5. Cachea dependencias en layers de Docker para builds rápidos
Jenkinsfile - Stage de Validación de Dependencias 
stage('Validar Dependencias') {
    steps {
        script {
            // Instalar en entorno limpio
            sh 'pip install --upgrade pip'
            sh 'pip install -r requirements.txt'
            
            // Verificar vulnerabilidades
            sh 'pip install safety'
            sh 'safety check --json --output safety-report.json'
            
            // Si hay vulnerabilidades críticas, fallar
            def report = readJSON file: 'safety-report.json'
            if (report.vulnerabilities.any { it.severity == 'high' }) {
                error("Vulnerabilidades críticas detectadas!")
            }
        }
    }
}
Seleccionado. Combina consistencia (Docker) + validación automática (Safety) + integración en pipeline (Jenkins). Cada commit pasa por este stage obligatoriamente.

Impacto Real en SISCAL

Antes (Manual) 3 incidentes/mes por dependencias desactualizadas o conflictivas
→
Después (Jenkins) 0 incidentes en 3 meses de operación. 100% de builds con deps validadas

Caso 3: Validación de Calidad de Código (Linting)

Escenario

Equipo de 5 desarrolladores trabajando en SISCAL. Cada uno con estilo de codificación ligeramente diferente. Necesitamos asegurar que todo el código siga estándares PEP 8 de Python antes de merge a main.

Alternativas Evaluadas

Opción A
Code Review Manual

Descripción: El líder técnico revisa manualmente cada Pull Request buscando problemas de estilo.

Problemas:

  • Subjetivo (qué es "código limpio")
  • Lento (30+ min por PR para revisar estilo)
  • Fatiga del revisor (después de 3 PRs, se pierde atención)
  • Inconsistente (reglas aplicadas diferente según día/humor)
Rechazado. No escala y consume tiempo de senior devs.
Opción B
Pre-commit Hooks Locales

Descripción: Instalar pre-commit hooks en la máquina de cada desarrollador que ejecuten flake8 antes de permitir commit.

Ventaja: Feedback inmediato, problemas detectados antes de push.

Problema: Depende de que cada desarrollador instale los hooks. Si uno no lo hace (olvido o intencional), código defectuoso llega al repo.

Solución parcial, no garantiza cumplimiento 100%.
Opción C
Jenkins Stage de Linting Obligatorio

Descripción: Stage 3 del pipeline ejecuta flake8 automáticamente. Si el código no cumple PEP 8, el pipeline falla y el deployment se bloquea.

Beneficios Clave:
  • Obligatorio: No se puede saltar. Todo código pasa por linting.
  • Objetivo: Mismas reglas para todos, sin interpretación humana.
  • Rápido: 30 segundos de ejecución automatizada.
  • Educativo: Desarrolladores aprenden estándares viendo errores.
  • Configurable: Archivo .flake8 versionado define reglas del equipo.
.flake8 - Configuración del Proyecto 
[flake8]
max-line-length = 120
exclude = .git,__pycache__,venv,build
ignore = E501,W503,E203

# E501: line too long (manejado por max-line-length)
# W503: line break before binary operator
# E203: whitespace before ':'
Seleccionado. Garantiza consistencia absoluta sin carga manual en code reviews. El equipo ahorra 2.5 horas/semana que antes dedicaban a revisar estilo.
Antes (Code Review Manual)
  • 30 min/PR para revisar estilo
  • 15 PRs/semana = 7.5 horas
  • Inconsistencias encontradas en producción
Ahora (Linting Automatizado)
  • 30 seg/PR automáticos
  • 15 PRs/semana = 7.5 minutos
  • 0 inconsistencias de estilo en producción

Caso 4: Rollback Rápido en Caso de Deployment Fallido

Escenario

Viernes 6:00 PM. Se despliega SISCAL v2.3.0 a producción. A las 6:15 PM, usuarios de Luz del Sur reportan que la funcionalidad de login falla. Necesitamos revertir a v2.2.9 (última versión estable) lo más rápido posible para restaurar servicio.

Alternativas Evaluadas

Opción A
Rollback Manual con Git

Proceso:

  1. Identificar commit de v2.2.9 en historial de Git (2 min)
  2. Hacer git checkout <commit-hash> (30 seg)
  3. Reconstruir imagen Docker (4 min)
  4. Detener contenedores actuales (30 seg)
  5. Levantar contenedores con código anterior (1 min)
  6. Verificar que funciona (2 min)
  7. TOTAL: ~10 minutos (si todo sale bien)

Riesgos:

  • Presión: Es viernes en la tarde, el equipo está nervioso
  • Error humano: Puede elegirse commit incorrecto bajo presión
  • Estado de BD: Si v2.3.0 hizo migraciones, rollback de código no revierte BD
  • Downtime: 10+ minutos sin servicio = clientes afectados
Rechazado. Demasiado lento y propenso a errores bajo estrés.
Opción B
Jenkins Rebuild de Build Anterior

Proceso:

  1. Abrir dashboard de Jenkins (10 seg)
  2. Identificar build #47 (v2.2.9, último exitoso) en historial (20 seg)
  3. Clic en "Rebuild" (5 seg)
  4. Jenkins automáticamente:
    • Hace checkout del commit correcto
    • Reconstruye imagen (con caché, 1 min)
    • Despliega contenedores
    • Verifica health checks
  5. TOTAL: ~2 minutos
Ventajas Críticas:
  • 87% más rápido: 2 min vs 10 min manual
  • Sin error humano: Jenkins usa el commit exacto del build anterior
  • Historial visual: Dashboard muestra qué builds fueron exitosos
  • Rollback de BD automático: Stage de backup permite restaurar BD si es necesario
  • Repetible: Si el rollback falla, puede reintentarse sin pensar
Seleccionado. En situaciones de emergencia, velocidad y confiabilidad son críticas. Jenkins proporciona ambas.

Caso Real en SISCAL

Fecha: 15 de Noviembre, 2025

Evidencia Técnica

Jenkinsfile Groovy 

pipeline {
    agent any
    
    environment {
        DOCKER_IMAGE = 'siscal-app'
        POSTGRES_CONTAINER = 'siscal-postgres'
    }
    
    stages {
        stage('Tests Unitarios') {
            steps {
                echo '========== ETAPA 4: Ejecutando tests =========='
                script {
                    if (isUnix()) {
                        sh 'pip install -r requirements.txt'
                        sh 'python3 -m pytest tests/ -v'
                    } else {
                        bat 'python -m pytest tests\\ -v'
                    }
                }
            }
        }
        
        stage('Construir Imagen Docker') {
            steps {
                echo '========== ETAPA 6: Construyendo nueva imagen =========='
                script {
                    // Build optimizado sin caché para asegurar limpieza
                    sh "docker-compose build --no-cache"
                }
            }
        }

        stage('Deploy a Producción') {
            when { branch 'main' }
            steps {
                echo '========== ETAPA 11: Deploy a Producción =========='
                // Despliegue automático solo si pasan todos los tests
                sh 'docker-compose up -d'
            }
        }
    }
}
